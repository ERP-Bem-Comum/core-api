# SOL-519 — Proposta de correção (READ-ONLY)

**Issue:** #519 — `fin_payables.status` / `fin_documents.status` `varchar(16)` não comportam
`'PartiallyReconciled'` (19 chars) → sob `sql_mode=STRICT_ALL_TABLES` o `UPDATE` aborta com
**errno 1406 / SQLSTATE 22001** e a transação de conciliação parcial (#141/#247) reverte inteira.

**Escopo:** módulo `financial` apenas (ADR-0014, um módulo por sessão). Sem `db:generate`, sem editar schema — só proposta.

---

## 0. Confirmação do defeito (ancorado nos arquivos reais)

| Item                               | Arquivo:linha                                        | Valor                                                                                              |
| :--------------------------------- | :--------------------------------------------------- | :------------------------------------------------------------------------------------------------- |
| Coluna `fin_documents.status`      | `schemas/mysql.ts:116`                               | `varchar('status', { length: 16 })`                                                                |
| Coluna `fin_payables.status`       | `schemas/mysql.ts:249`                               | `varchar('status', { length: 16 })`                                                                |
| CHECK `fin_documents_status_chk`   | `schemas/mysql.ts:184-187`                           | `IN ('Draft','Open','Approved','Transmitted','Refused','Paid','PartiallyReconciled','Reconciled')` |
| CHECK `fin_payables_status_chk`    | `schemas/mysql.ts:272-275`                           | idem                                                                                               |
| Origem do valor                    | `domain/payable/reconciled-status.ts:16`             | `reconciledSumCents >= valueCents ? 'Reconciled' : 'PartiallyReconciled'`                          |
| Gravação                           | `repos/reconciliation-repository.drizzle.ts:100-109` | `tx.update(finPayables).set({ status: nextStatus })`                                               |
| Migration do CHECK contraditório   | `migrations/mysql/0024_mysterious_the_spike.sql:3-4` | `ADD CONSTRAINT ... IN (...,'PartiallyReconciled',...)`                                            |
| Tipo de domínio (fonte da verdade) | `domain/document/types.ts:34-42`                     | `DocumentStatus` = 8 literais, maior = `'PartiallyReconciled'`                                     |

**Contradição estrutural confirmada:** a migration 0024 gravou um CHECK que _lista_ um literal
(`'PartiallyReconciled'`, 19 chars) que a própria coluna (`varchar(16)`) não comporta. O CHECK aceita,
o tipo da coluna rejeita — e sob STRICT o rejeito vence com 1406.

Só o caminho **parcial** dispara: uma conciliação **total** grava `'Reconciled'` (10 chars, cabe em 16);
a parcial grava `'PartiallyReconciled'` (19). Por isso o CA11 de
`reconciliation-repository.drizzle-mysql.test.ts:165` (que testa reconciliação total) nunca pegou o bug.

## 1. Largura correta — **`varchar(24)`**

### Contagem dos literais (maior de cada CHECK das duas tabelas)

`Draft`(5) `Open`(4) `Approved`(8) `Transmitted`(11) `Refused`(7) `Paid`(4) **`PartiallyReconciled`(19)** `Reconciled`(10).

**Piso absoluto = 19.** Abaixo disso o schema se contradiz.

### Proposta: `varchar(24)` (folga sã, não exagerada)

Justificativa (em ordem de peso):

1. **Consistência com a família de enums do próprio `fin_`.** O módulo já padroniza os enums "largos" em
   `varchar(24)`: `fin_documents.payment_method`/`fin_payables.payment_method` (`schemas/mysql.ts:100,258`,
   maior literal `'TransferenciaBancaria'`=21) e `fin_manual_entries.type` (`:800`, `'FeePenaltyInterest'`=18).
   `status` 16→24 alinha ao número "de família" do schema, não inventa um valor ad-hoc.
2. **Evita o anti-padrão de encaixe exato.** Encaixe justo (`varchar(19)`) reintroduz o defeito: qualquer
   status futuro > 19 quebra de novo. O próprio schema **já sofre** desse anti-padrão em
   `contracts.ctr_documents.status varchar(16)` com `'LogicallyDeleted'`=exatos 16 (§4). A nota da issue pede
   explicitamente "derivar a largura do maior literal aceito pelo CHECK, para o schema não voltar a se
   contradizer" — 24 dá 5 chars de margem sem desperdício.
3. **Custo de armazenamento nulo.** No InnoDB o `VARCHAR(N)` cobra pelo conteúdo real + 1 byte de prefixo
   (16 e 24 estão ambos na faixa ≤255 bytes com utf8mb4: 16×4=64B, 24×4=96B → 1 byte de comprimento). 19, 20
   e 24 têm **exatamente o mesmo custo** por linha. O índice (`fin_payables_status_idx` / `fin_documents_status_idx`)
   sobre `varchar(24)` utf8mb4 = 96 bytes de chave, muito abaixo do limite de 3072.

**Ancoragem ADR:** ADR-0020 §"Mapeamentos" (`0020...md:69-70`) e ADR-0018 (`0018...md:72`) fixam o padrão
`VARCHAR(N) + CHECK` para enums (ENUM nativo proibido — `0020...md:104`, `0018...md:101`). O `16` daqueles
exemplos é a largura que **coincidentemente** cobria os statuses de Contracts; não é uma constante mágica.
Para `DocumentStatus` (8 valores, maior 19), a largura correta é ≥19 → 24 pela regra de família.

> **Alternativa mínima:** `varchar(20)` (margem de 1). Aceitável e mais "enxuto", mas quebra a família `fin_`
> (nenhum outro enum do módulo usa 20 para status) e dá 1 char de folga. **`varchar(19)` deve ser rejeitado**
> (encaixe exato = o anti-padrão que originou #519). Recomendo **24**; deixo a decisão final ao P.O./revisor.

## 2. A migration — como gerar e o SQL esperado

### Como gerar

Alterar **só a largura** nas duas colunas do schema (`length: 16` → `length: 24` em `:116` e `:249`) e rodar:

```bash
pnpm run db:generate:financial   # drizzle-kit generate --config db/drizzle/financial.ts
```

(config: `db/drizzle/financial.ts` → out `src/modules/financial/adapters/persistence/migrations/mysql/`).
Próxima migration = `0039_*.sql`. **Depois:** `prettier --write` nos `meta/*.json` gerados (o hook reformata
pós-commit — memória `prettier-hook-reformats-after-commit`).

### SQL que o drizzle-kit deve emitir

```sql
ALTER TABLE `fin_documents` MODIFY COLUMN `status` varchar(24) NOT NULL;--> statement-breakpoint
ALTER TABLE `fin_payables`  MODIFY COLUMN `status` varchar(24) NOT NULL;
```

### ⚠️ Restrição crítica MySQL 8.4 — SEM hint de algoritmo (o `ALTER` cai para COPY)

- **Refman 8.4, Table 17.17** (`17-innodb-storage-engine.part02.md:2051`):
  `Extending VARCHAR column size — Instant: No · In Place: Yes · Rebuilds Table: No · Permits Concurrent DML: Yes · Only Modifies Metadata: Yes`.
- **Refman §"Extending VARCHAR column size"** (`:2182-2194`): _"The number of length bytes required by a VARCHAR
  column must remain the same. For VARCHAR columns of 0 to 255 bytes in size, one length byte is required...
  in-place ALTER TABLE only supports increasing VARCHAR column size from 0 to 255 bytes..."_.
  16→24 (utf8mb4: 64B→96B, ambos ≤255B → 1 byte de comprimento inalterado) é, **em teoria**, INPLACE/metadata-only.
- **PORÉM — divergência verificada nesta versão** (memória `mysql84-alter-varchar-no-algorithm-hint`, validada no
  fix #274 contra `mysql:8.4.9`): forçar `ALGORITHM=INPLACE` **ou** `ALGORITHM=INSTANT` no widening de VARCHAR
  retorna **ERROR 1845 (0A000)** — em utf8mb4 **e** latin1 (não é charset). Contradiz a Table 17.17. O `ALTER`
  **sem hint** funciona: o servidor cai para o melhor disponível, que nessa versão é **`ALGORITHM=COPY`**.
- **Conclusão:** o drizzle-kit gera o `MODIFY COLUMN` **sem** hint por padrão — deixar assim. **NÃO** adicionar
  `ALGORITHM=INPLACE/INSTANT` manualmente: passaria os gates unit (que não aplicam migration) e só quebraria com
  1845 no apply real (CI integration ou prod) — exatamente o modo de falha que a memória alerta.

### Impacto de lock em PRODUÇÃO (tabela com dados)

`ALGORITHM=COPY` é _table-rebuilding_ (Refman `:1684`) e, por padrão, roda sob `LOCK=SHARED` (Refman `:1897`):
**leituras concorrentes permitidas, ESCRITAS bloqueadas** por uma duração **∝ nº de linhas** (cópia integral +
reconstrução de índices e FKs).

- **Contagem de linhas de `fin_documents`/`fin_payables` em prod: desconhecida** (não medida aqui). Risco genérico:
  se as tabelas forem grandes/quentes, todo `INSERT/UPDATE/DELETE` de documentos e títulos **pausa** durante a cópia.
  `fin_payables` tem FK `fin_payables_document_id_fk` ON DELETE CASCADE → a reconstrução também revalida a FK.
- **Mitigações** (memória `mysql84-alter-varchar-no-algorithm-hint` + `validate-mysql-always-x99`):
  1. Aplicar em **janela de manutenção / baixo tráfego** (o writer pool aplica migration no boot — coordenar deploy).
  2. Se as tabelas forem grandes, usar **pt-online-schema-change / gh-ost** (online real) em vez de confiar no COPY.
  3. **Validar o `ALTER` real** num MySQL 8.4 da mesma versão (x99/VM) com dataset com formato de prod **antes do merge** —
     não confiar só no Refman nem só nos gates unit.
- **Provável nesta fase:** ERP greenfield, volumes baixos (milhares, não milhões) → COPY é um blip curto. Mas a
  proposta trata o risco genericamente; a decisão de janela depende da contagem real medida em prod/QA.

## 3. O CHECK precisa mudar? **Não.**

- **Widening não invalida o CHECK.** `MODIFY COLUMN ... varchar(24)` só _aumenta_ a capacidade; todos os valores
  existentes continuam satisfazendo `status IN (...)`. O CHECK `fin_*_status_chk` permanece válido e ativo.
- **O drizzle-kit não deve re-emitir o CHECK.** Como os literais do CHECK no schema não mudam, o diff do kit
  contém **apenas** os dois `MODIFY COLUMN`. ⚠️ **Auditar mesmo assim:** o drizzle-kit para MySQL às vezes emite
  `DROP CONSTRAINT` + `ADD CONSTRAINT` espúrio para CHECKs (foi o que a 0024 fez). Se reaparecer DROP+ADD com os
  **mesmos literais**, é inócuo; se aparecer **só DROP** (sem re-ADD), é bug — corrigir à mão. Revisar linha-a-linha
  antes de commitar (regra do agente: `db:generate` é confirmação, não descoberta).
- **CHARSET/COLLATE:** `status` **não** é coluna UUID → usa o collate default da tabela (`utf8mb4_unicode_ci`),
  não `utf8mb4_bin`. `MODIFY COLUMN` sem cláusula `COLLATE` herda o default da tabela → **sem drift**. Confirmar
  no SQL emitido; se quiser blindar, anexar `COLLATE utf8mb4_unicode_ci` explícito ao MODIFY (opcional).

## 5. Prova de W0 (teste RED contra MySQL real)

**Onde vive:** `tests/modules/financial/adapters/persistence/payable-status-width.drizzle-mysql.test.ts`
(novo arquivo, ao lado de `reconciliation-repository.drizzle-mysql.test.ts`).

**Por que um teste de persistência dedicado, e não reusar `realized-provisioned` CA2/CA4:** o CA2/CA4 falha pelo
bug **incidentalmente** (via projeção). Um teste focado no nível do adapter pina **o defeito exato** (a largura da
coluna) e é imune a churn na projeção/reader. Complementa — não substitui — o CA11 existente (que só cobre total).

**Harness** (molde de `reconciliation-repository.drizzle-mysql.test.ts:46`): gate `if (!process.env['MYSQL_INTEGRATION'])`
→ `describe.skip`; roda em `pnpm run test:integration:financial`. Fora do gate, roda no `pnpm test` puro apenas a
parte estrutural (sem DB).

**Formato do RED** (o que DEVE falhar hoje com 1406, e passar após o widen):

```
// 1. INSERT de um fin_payables 'Paid' válido (varchar(16) cabe 'Paid').
// 2. UPDATE fin_payables SET status='PartiallyReconciled' WHERE id=?   // ← estoura 1406/22001 HOJE
// 3. read-back: SELECT status → assert.equal(row.status, 'PartiallyReconciled')
// (idem para fin_documents)
```

Asserção do RED: em W0 o passo 2 rejeita com `errno: 1406, sqlState: '22001'` (`Data too long for column 'status'`) —
capturável via `e.cause.errno` no `DrizzleQueryError` (memória `drizzle-execute-error-cause-errno`). Esse é o RED
pelo motivo certo (a coluna curta), não asserção frouxa.

**Critério de aceite ligado (issue #519):** após o widen + migration, `pnpm run test:integration:financial` verde,
incluindo CA2/CA4 de `realized-provisioned` (que voltam a passar de brinde).
